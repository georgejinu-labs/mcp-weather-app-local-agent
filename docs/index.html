Context Window & Tokens

What is a token?
DEFINITION

A token is a chunk of text — roughly 0.75 words on average. The LLM doesn't see letters or words — it sees tokens. Every prompt you send and every response you get is measured in tokens.

"What is the weather in Tokyo?"

  What | is | the | weather | in | Tokyo | ?
  tok1   tok2   tok3    tok4      tok5    tok6   tok7
  ≈ 7 tokens   (roughly 1 token per word, punctuation counts too)

Rule of thumb:
  100  tokens ≈  75 words ≈ half a page of text
  1000 tokens ≈ 750 words ≈ a short article
What is the context window?
DEFINITION

The context window is the LLM's working memory — everything it can "see" at once when generating a response. It includes the system prompt, full conversation history, tool results, and your current question. Once full, old content gets dropped.

Think of it like a whiteboard with limited space:

  ┌─────────────────────────────────────────────────┐
  │  CONTEXT WINDOW  (e.g. 32,768 tokens max)       │
  │                                                 │
  │  [System prompt + tool list]   ~300 tokens      │
  │  [Turn 1 human question]       ~20  tokens      │
  │  [Turn 1 AI tool call]         ~50  tokens      │
  │  [Tool result injected]        ~81  tokens      │
  │  [Turn 1 AI final answer]      ~96  tokens      │
  │  [Your new question]           ~10  tokens  ← just added │
  └─────────────────────────────────────────────────┘
  Total used: ~557 tokens. Still 32,211 tokens of room.

Input tokens vs output tokens — why this matters for cost
Every LLM call has TWO token counts:

  Input tokens  = everything the LLM READS
                  (system prompt + history + tool list + your question)
                  Charged at ~$3 per million tokens (Claude Sonnet)

  Output tokens = everything the LLM WRITES
                  (tool calls + final answer)
                  Charged at ~$15 per million tokens (Claude Sonnet)

From your trace:
  Call 1:  input=393  output=73   total=466
  Call 2:  input=474  output=96   total=570

With Ollama locally → FREE. No token cost at all.
With Claude API in prod → small cost, very predictable.

This is why long conversations cost more — input grows with every turn.
What happens when you hit the token limit?
ScenarioWhat happensFix
Context overflow Oldest messages are silently dropped. LLM "forgets" early conversation. Summarize old turns or use external memory (DynamoDB)
API error Some APIs throw an error instead of truncating. Sliding window — keep only last N turns in context
Slow responses Larger context = more tokens to process = slower output. Trim tool results before injecting (summarize, not raw data)

max_steps=15 — The safety brake on the loop

What is one "step"?

One step = one full loop iteration: LLM thinks → (optionally calls a tool) → result fed back. Without a limit, a confused LLM could loop forever. max_steps is the hard brake. 

Your trace had 2 steps for "What is the weather in Tokyo?":

Step 1:  LLM Call #1  → decided to call get_weather
         Tool call     → get_weather(city=Tokyo) → got result
         ModelCallLimitMiddleware: run_model_call_count = 1

Step 2:  LLM Call #2  → read result, wrote final answer
         ModelCallLimitMiddleware: run_model_call_count = 2
         tool_calls=[] → DONE. Exited loop. Used 2 of 15 steps.
Normal exit vs max_steps hit
Normal exit — LLM finishes early
1 call tool A → result
2 call tool B → result
3 final answer ✓ DONE
tool_calls=[] → loop exits cleanly. Full answer returned.
max_steps hit — forced stop
1 call tool A → error, retrying...
2 call tool A → error, retrying...
· · · keeps looping · · ·
15 HARD STOP ✗
StopIteration raised. Returns partial result or empty string.
What causes an agent to hit max_steps
1. Tool keeps failing → LLM keeps retrying
   get_weather → timeout → retry → timeout → retry...

2. LLM gets confused in a loop
   calls tool A → gets result → calls tool A again (same params!)

3. Task genuinely needs more steps than you allowed
   PA agent processing 5 requests but max_steps=10
   → runs out before finishing all 5

4. LLM over-planning
   breaks a simple task into 20 tiny tool calls when 5 suffice
Recommended max_steps for your PA agent + how to handle the hit
PA agent steps per request:
  Step 1: query PA database           → pending request data
  Step 2: check member eligibility    → eligibility API
  Step 3: search clinical guidelines  → guidelines DB
  Step 4: evaluate criteria           → LLM reasons (no tool call)
  Step 5: update PA status in DB      → write result
  Step 6: generate approval letter    → document tool
  Step 7: send notification           → fax/email tool
  Step 8: LLM writes final summary    → DONE

max_steps=15 gives buffer above the ~8 steps needed.
Too low (e.g. 5)    → agent stops mid-workflow.
Too high (e.g. 1000) → stuck agent burns tokens forever.

Handle it in your code (Week 4 human-in-loop gate):
try:
    response = await agent.run(f"Process PA request {pa_id}")
    if not response or len(response.strip()) < 20:
        await flag_for_human_review(pa_id, reason="agent_incomplete")
except Exception as e:
    await flag_for_human_review(pa_id, reason=str(e))

LangChain and LangGraph — where they fit

Your understanding — confirmed and sharpened
✓ CORRECT: "LangChain is a framework for common message formats
               (System, Human, AI, Tool messages)"

✓ CORRECT: "LangGraph is the loop"

→ ADD THIS: Nodes belong to LangGraph — they ARE the loop.
               LangGraph = loop engine + nodes + edges.
               A node IS a named step in the loop. Not a separate thing.

Where do nodes live?
  Not in LangChain.   LangChain has no concept of nodes.
  Not in the LLM.     The LLM has zero awareness of nodes.
  Only in LangGraph.  Nodes are a LangGraph concept, full stop.
Each library owns exactly one concern
LangChain message formats + interfaces
Owns: SystemMessage, HumanMessage, AIMessage, ToolMessage
Owns: StructuredTool — the format mcp-use converts MCP tools into
Owns: the standard .invoke(messages) interface every LLM adapter uses
Does NOT own: nodes, edges, loops, graphs — none of that.
LangGraph loop + nodes + edges
Owns: nodes — named Python functions registered as steps
Owns: edges — routing rules between nodes ("if tool_calls → tools node, else → END")
Owns: state — the growing messages list passed between nodes
Owns: the loop itself — run node → check edge → run next node → repeat
Uses LangChain message types (HumanMessage etc.) internally.
LLM — qwen3.5:9b via Ollama just reads + writes text
Zero awareness of nodes, edges, or loops.
Just receives a prompt (text) → returns a response (text or tool call JSON).
It runs inside the model node. It has no idea it's in a graph.
A node is literally just a Python function — registered with LangGraph
# model node — just a function
def model_node(state):
    messages = state["messages"]          ← reads current state (LangChain messages)
    response = llm.invoke(messages)       ← calls the LLM (LangChain interface)
    return {"messages": [response]}       ← updates state with AIMessage

# tools node — just a function
def tools_node(state):
    tool_call = state["messages"][-1]     ← reads LLM's tool_call from state
    result = execute_tool(tool_call)      ← mcp-use fires call_tool on server.py
    return {"messages": [result]}         ← updates state with ToolMessage

# LangGraph wires them together with edges:
graph = StateGraph(MessagesState)
graph.add_node("model", model_node)     ← register node 1
graph.add_node("tools", tools_node)     ← register node 2

graph.add_edge(START, "model")
graph.add_conditional_edges(              ← the routing rule (the edge)
    "model",
    lambda state: "tools" if state["messages"][-1].tool_calls else END
)
graph.add_edge("tools", "model")       ← always back to model

# mcp-use builds this entire graph for you automatically.
# You just call agent.run() and it runs the whole thing.

ModelCallLimitMiddleware.before/after_model = extra middleware nodes
mcp-use adds these automatically to enforce max_steps=15.
Why nodes instead of a plain while loop?
You COULD write the loop yourself (and it works):
while True:
    response = llm.invoke(messages)
    if response.tool_calls:
        result = execute_tool(response.tool_calls[0])
        messages.append(result)
    else:
        break

But LangGraph nodes give you things a plain loop cannot:

  Visibility:    every node logs input/output  ← you saw this in your trace
  Middleware:    inject logic between nodes     ← how max_steps counter works
  Branching:    route to different nodes       ← error node, retry node etc.
  Checkpoints:  pause and resume at any node   ← your Week 4 human-in-loop gate!
  Parallelism:  run multiple nodes at once     ← multi-agent setups
LangChain
The common language. HumanMessage, ToolMessage, StructuredTool, ChatOllama — all LangChain types.
LangGraph
The loop engine. model node → tools node → model node, with state, edges, and step counting. Built on LangChain.
Ollama / LLM
Just the brain. Reads prompt, writes response. No awareness of nodes, graphs, or loops.
